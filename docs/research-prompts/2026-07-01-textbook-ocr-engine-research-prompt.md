# 教科书/扫描书 OCR 引擎选型调研 —— 网页端 Claude 提示词

- 日期：2026-07-01
- 用途：为仓库待建的 `scripts/pipelines/textbooks/` 管线做引擎选型调研
- 执行方式：**把下方「提示词正文」整块复制到网页端 Claude（开启联网 / Research 能力）运行**
- 产出：一份引擎选型**综述报告**（文献级，非本地实测）；本地实测是后续在家用机另做的一条线

## 使用说明（给所有者，不用复制进 Claude）

- 网页端 Claude 只能**联网检索 + 综合**，看不到本工作区、不能下载引擎跑你的样本。所以提示词已**自带全部场景约束**，并明确要求它**不得编造实测数字**。
- 若你手头有「Unlimited OCR」的确切链接/来源，运行前贴到提示词末尾，比让它盲查更稳（当前无法在已知引擎中对上号，见正文 §候选）。
- 报告拿回来后，进入下一条线：在家用机对它推荐的 Top 1-2 引擎跑你的扫描样本做本地实测。

## 提示词正文（复制以下全部）

````text
你是文档转换与 OCR / 文档版面解析（document layout parsing）引擎的选型专家。请为我做一次引擎选型调研，产出一份**选型综述报告**。

## 你的能力边界（必须诚实遵守）
- 你可以联网检索并阅读官方文档、论文、benchmark、GitHub repo、可信第三方评测与社区实测，然后综合。
- 你**不能**在我的样本上实测。因此：凡涉及准确率 / 质量的结论，必须基于**公开可查的证据**（benchmark、论文、官方评测、可信第三方或社区实测），并注明出处与测试条件。
- **严禁编造实测数字或假装你亲自测过。** 无可靠数据处，直接写"缺公开数据，需本地实测验证"。
- 引擎迭代很快：优先采用**近 6–12 个月**的信息，对每个关键结论标注信息时点（如"截至 2026-06"），并区分"官方宣称"与"第三方独立验证"。

## 场景画像（选型必须紧贴，别泛泛而谈）
目标：把一大批**教科书 / 扫描书**转成高质量 **Markdown**（供人阅读 + 进知识库）。

内容特征：
- 约 **63% 是纯扫描件、无文本层**（image-only PDF）——需要真正的 OCR，不是抽取已有文本层。
- 大量**中文（CJK）**内容，且**中英混排**。样例：Pozar《微波工程》中英文版、《高等数学》、Khalil《Nonlinear Systems》。
- **数学公式密集**：上下标、希腊字母、分式、矩阵、多行公式、积分 / 求和等，期望输出为 **LaTeX**。
- **双栏 / 多栏**排版、图文混排、表格、插图带 caption。
- 大部头：单本常 **700+ 页**，总量约 **3.6 万扫描页**。

部署约束（作可行性过滤）：
- 优先能**本地离线**运行；GPU 为 **RTX 5060 / 8GB 显存**，CPU 可回退但慢。
- 云 API 不是首选，但可纳入作为"效果天花板"参照。

**成本（重要，因为量大）**：总量约 **3.6 万页**，规模不小，**价格是真实决策因素**。请认真评估：
- 本地开源方案的成本结构（基本是一次性硬件 + 电费，按量边际成本≈0）；
- 云 API 方案按量计费时，**在 3.6 万页规模下的总成本量级估算**（列出单价来源与计价单位：按页 / 按 token / 按调用），让我能直观看到"走云要花多少钱"。

许可证：**不是**主要决策因素，仅在报告里附带列出（类型、是否商用友好），不要用它做筛选门槛。

## 评估重心（严格按此优先级，前者远重于后者）
1. **转换效果 / 质量（第一位，报告主体）**：
   a. 纯扫描件文字识别准确率，尤其**中文 CJK**（给 CER / 准确率数据 + 出处）；
   b. **数学公式 → LaTeX** 的正确率与复杂公式处理能力；
   c. **版面分析与阅读顺序**：双栏正确串接、图文 / 表格位置、多栏阅读序；
   d. 表格结构还原、插图与 caption 关联、中英混排、特殊符号；
   e. 对扫描质量（倾斜、噪点、低分辨率、老书影印）的鲁棒性。
2. **工程可行性与成本（第二位，量大所以成本权重不低）**：
   a. 工程：8GB 显存能否跑 / 需多少显存、速度与吞吐（页 / 分钟量级）、700+ 页大文件的分块支持、是否支持批处理与断点续跑；
   b. 成本：本地方案的边际成本（≈0）vs 云方案在 **3.6 万页规模下的总价估算**（注明单价来源与计价单位）。**这是重要维度，不是附注。**
3. **附注信息（列出即可）**：许可证、维护活跃度、社区规模、最近更新时间。

## 候选引擎（逐一评估，可增补你发现的其他强候选）
- **本地开源（正选）**：MinerU 2.x、PaddleOCR-VL / PP-StructureV3、DeepSeek-OCR、olmOCR / olmOCR 2、dots.ocr、GOT-OCR 2.0、Marker / Surya、Nanonets-OCR、docTR（若仍相关）。
- **云 API（仅作"效果天花板"参照，不参与本地排名）**：Google Gemini（文档理解）、Mistral OCR、Google Document AI、Azure Document Intelligence，以及其他你认为代表当前 SOTA 的。
- **特别核查**：是否存在一个名为 **"Unlimited OCR"** 的引擎 / 产品？若有，评估它；若查不到，明确说明未找到（可能是别名或名称记忆有误），不要牵强附会。

## 核心判断题（报告必须给出有依据的结论，不能只罗列）
1. **架构路线**：对教科书场景，应该选"引擎**只输出结构化中间产物**（版面 bbox + 阅读顺序 + 文本块 JSON），再由下游自建确定性逻辑重建 Markdown"，还是"直接采用引擎**端到端吐出的 Markdown**"？两条路线在**质量 / 可控性 / 可审计性 / 出错可定位性**上的权衡分别是什么？给出你的建议及理由。
2. **单引擎 vs 组合管线**：一个端到端引擎够用，还是需要"版面模型 + OCR 模型 + 公式识别模型"的组合？若推荐组合，给出具体的组合方案示例。
3. **收敛推荐**：综合转换效果，给出 **Top 2–3 个本地开源候选**，各自的适用边界与已知短板；并明确**推荐下一步优先本地实测哪 1–2 个**、用什么样本、看哪些指标来验证。

## 输出格式（Markdown 综述报告）
1. **执行摘要**：核心结论 + Top 推荐 + 一句话理由。
2. **候选横向对比大表**，列建议为：引擎 / 类型（本地 or 云）/ CJK 准确率 / 公式→LaTeX / 版面阅读顺序 / 表格 / 8GB 可行性 / 速度量级 / **3.6 万页成本量级** / 许可 / 最近更新 / 关键出处。
3. **每个候选一节**：能力、教科书场景适配度、短板、可信出处。
4. **三个核心判断题**的逐一结论。
5. **本地方案 vs 云 SOTA：效果 × 成本双维度对比**——既让我知道本地方案距 SOTA 效果差多远，也算清"3.6 万页若走云 API 的总花费量级 vs 本地几乎零边际成本"，帮我判断这个效果差价不值得。
6. **推荐的下一步本地实测方案**：选哪个、测什么样本、看哪些指标。
7. **参考来源清单**：全部带链接，标注信息时点与可信度层级（官方 / 论文 / 第三方评测 / 社区）。

## 硬性要求
- 所有质量 / 准确率结论必须标注出处；无出处的推断显式标注"[推断，待实测]"。
- 涉及具体数字（CER、显存、速度）时，注明测试条件（数据集、语言、硬件），否则说明条件未知。
- 结论要服务于"教科书 + 中文 + 公式 + 扫描件 + 本地 8GB"这个具体场景，不要给通用泛泛结论。
````
