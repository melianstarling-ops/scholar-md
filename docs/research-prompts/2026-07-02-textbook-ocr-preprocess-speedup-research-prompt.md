# 教科书 OCR 图像预处理 + 本地推理加速 —— 网页端 Claude 提示词

- 日期：2026-07-02
- 用途：`textbooks` 管线首版已建成（引擎 PaddleOCR-VL 1.6），准备对 800 页大部头样本做大文件稳健化
  测试前，本机（RTX 5060）装环境实测发现 **GPU 架构不兼容、直接跑不起来**（`paddlepaddle-gpu` cu126
  稳定版没编译 sm_120 kernel）——上线前需要搞清楚两件事：① 这个 GPU 兼容性问题怎么解；② 顺带查清楚
  有没有便宜的图像预处理能提升识别质量、本地推理还能怎么提速。
- 执行方式：把下方「提示词正文」整块复制到网页端 Claude（开启联网 / Research 能力）运行
- 产出：一份**可执行的调研报告**（文献级，非本地实测）；本地实测是后续在家用机另做的一条线

## 我的具体场景（必须紧贴，不要给通用泛泛结论）

### 当前管线现状
- 引擎：`PaddleOCR-VL 1.6`（Python 包 `paddleocr[doc-parser]==3.7.0`），本地直调
  `PaddleOCRVL(pipeline_version="v1.6").predict(png_path)`，**单页同步调用、无批处理、无服务化**。
  惰性单例（进程内只加载一次模型）。
- 预处理：仅 PyMuPDF `get_pixmap(dpi=200)` 把 PDF 页栅格化成 PNG，**没有任何图像质量处理**
  （无去噪、无二值化、无倾斜矫正 deskew、无对比度增强），加这一步纯粹是因为扫描 PDF 直接喂引擎会在
  图像解码环节崩溃，跟质量无关。
- 是否已启用 PaddleOCR-VL 内置的文档方向分类（doc orientation classify）/ 文本图像展平（unwarping）
  子模块**未知未验证**——调用时没有显式传任何相关参数，实际生效状态待查（这也是你要查的一部分：
  这些参数默认值是什么、去哪个配置文件/API 能查/改）。
- 内容特征：中英混排扫描教科书（电磁学/微波工程/数学方向，如 Pozar《微波工程》、Paul《Analysis of
  Multiconductor Transmission Lines》、Jackson《经典电动力学》），双栏排版，公式密集（要求输出 LaTeX），
  部分书籍老旧影印、可能有轻微倾斜/污渍/低对比度。
- 规模：单本常 700+ 页，总量约 3.6 万扫描页。

### 硬件（两台机器交替开发，方案要覆盖两者）
- 机器 A：Windows 11 + AMD Ryzen 9 9900X + NVIDIA GeForce **RTX 5060 8GB**（Blackwell 架构，
  驱动上报 CUDA 13.1，`nvidia-smi` 显示 Driver Version 591.86）。
- 机器 B：Windows 11 + AMD Ryzen 9 5900 + NVIDIA GeForce **RTX 4060 8GB**（Ada Lovelace 架构）。
- Python 环境：独立 venv，`paddlepaddle-gpu==3.2.1`（官方 **cu126** 索引安装）+
  `paddleocr[doc-parser]==3.7.0`。
- **【已实测确认，非推测】cu126 稳定版 wheel 在 RTX 5060 上直接跑不起来**：装好后首次调用
  `PaddleOCRVL(pipeline_version="v1.6")` 报错
  `RuntimeError: Unsupported GPU architecture`，日志明确写
  `the installed PaddlePaddle package was compiled for 75 80 86 89, but your current GPU is 120`。
  即该 wheel 编译时只打包了 sm_75/80/86/89（Turing/Ampere/Ada）的 kernel，**完全没有 sm_120
  （Blackwell，RTX 50 系）**，不是降级/JIT 兼容跑得慢，是**直接抛异常起不来**、连一页都跑不了。
  **这是本次调研最需要给出可执行答案的问题**：官方是否有已发布的 cu128/cu129 或更新索引的 wheel
  已经带 sm_120 支持？如果稳定版没有，`develop`/nightly 分支构建是否可用（给出安装命令与已知坑）？
  若两者都不可行，除了换到 sm_89 及以下架构的 GPU（如 RTX 4060），还有没有别的路径（比如降级用
  ONNX Runtime GPU 后端绕开 Paddle 原生 kernel，或用 Docker 里的官方镜像）？
  RTX 4060（Ada，sm_89）在同一 wheel 上是否能正常跑（同项目另一台机器用的是这块卡，
  按 wheel 编译清单里含 89，理论可行，但也请一并确认有无其他已知坑）。
- 因为 GPU 推理直接跑不起来，**"70 秒/页"这个数字来自更早一次会话的记忆、非本次实测，不保证仍准确、
  也不确定是 GPU 还是 CPU 路径测的**——请不要以此为基准数字展开分析，只作为"背景印象"参考；
  Part B 的诊断结论应该建立在"当前 GPU 架构不兼容、需先解决兼容性才能拿到真实速度"这个前提上。

### 已知的加速候选方向（帮我逐一评估可行性，不要漏掉，也欢迎补充我没想到的）
1. PaddleOCR/PaddleX 的"高性能推理"选项（如启用 TensorRT / ONNX Runtime GPU / MKLDNN 等后端插件）。
2. Batch 推理：`.predict()` 是否支持一次传入多张图/一个目录，内部自动 batching，而不是我们现在的
   逐页 for 循环单独调用。
3. 精度：FP16 / INT8 量化对这个模型的速度提升与质量损失的权衡。
4. 服务化部署：PaddleOCR-VL 官方的 `genai_server`（FastDeploy / vLLM 风格）支持 `max-num-seqs`
   并发请求处理——这条路线相比我们现在的本地直调 `.predict()`，吞吐提升的量级大概是多少？部署复杂度多高？
   值不值得为本地单机场景（非多用户服务）引入。
5. **vLLM 加速**：本项目已知 vLLM 在 Windows 无原生支持，需 WSL2 或 Docker。请评估：WSL2 跑
   vLLM/PaddleOCR-VL 服务化，相比原生 Windows 直调，速度提升量级、配置复杂度、8GB 显存下的可行性。
6. 简单多进程/多实例：单卡 8GB 下开 2 个进程各自加载模型（单实例 FP16 基线显存约 2.1GB，来源见你
   查到的官方文档）是否现实？GPU 计算本身是否真能并行，还是只是任务排队、吞吐提升有限甚至负收益
   （本项目在 CPU OCR 场景上曾实测"进程数越多反而越慢"，GPU 场景下这个风险是否依然存在，请给出依据）。
7. 分辨率/DPI 权衡：当前固定 200dpi 栅格化，更低 DPI（如 150）对这个模型的速度提升与识别质量损失
   分别是多少量级（有公开数据的话）。

## 评估重心

### Part A：图像预处理提升 OCR 质量
1. **关键问题**：PaddleOCR-VL 这类 VLM/多模态文档解析模型，跟传统 OCR（Tesseract 类）不同——
   它是在自然图像/文档截图上训练的。经典 CV 预处理手段（二值化、激进降噪、高对比度锐化）对
   **传统 OCR 有效，但对 VLM 类模型是否反而有害**（把图像变得"不自然"，偏离训练分布）？
   请给出你查到的证据或业界共识，明确回答：对我们这个引擎，哪些预处理**该做**、哪些**不该做**。
2. 对老旧扫描教科书常见问题（轻微倾斜、页面弯曲/透视变形、纸张泛黄导致低对比度、装订阴影、
   摩尔纹）逐项给出：值不值得处理、用什么开源工具/算法（如 OpenCV deskew、`docTR`/`unpaper`
   风格的页面净化）、对最终识别准确率的预期影响量级（有数据则给数据，无则标注"待实测"）。
3. **DPI 选型**：200dpi 是否是该模型的合理输入分辨率？过高/过低分别的影响，模型是否有推荐的
   输入分辨率区间（很多 VLM 内部有固定或建议的图像 tile/resize 逻辑，输入分辨率超出范围可能被
   自动降采样、白费栅格化的高 DPI）。
4. PaddleOCR-VL 内置的文档方向分类/展平子模块：默认是否开启、如何显式控制、值不值得依赖它而不
   自己做 deskew。

### Part B：本地推理加速
1. **第一优先级（阻塞性问题，必须先给出可执行答案）**：RTX 5060（sm_120）在 `paddlepaddle-gpu==3.2.1`
   cu126 稳定版下直接 `RuntimeError: Unsupported GPU architecture`，完全跑不起来（见上方"已实测确认"）。
   给出具体可执行的解法路径（官方 cu128/cu129 wheel？develop/nightly 构建？ONNX Runtime GPU 后端？
   Docker 官方镜像？），每条路径附安装命令来源与已知风险；如果确认"稳定版目前就是不支持，只能等官方
   发版或换 GPU"，也要明确说清楚，不要含糊。
2. GPU 兼容性问题解决、能正常跑起来之后，针对"单页/多页实测耗时"给出**诊断优先级清单**：先查什么
   （是否真的用了 GPU 而非回落 CPU、是否用了正确的 CUDA/cuDNN 版本、是否 FP32 而非 FP16）、后调什么。
   这部分不要求你有具体秒数结论（你测不了我的硬件），只要给出"我该怎么测、测出来怎么判断正常与否"的方法论。
3. 对上面 7 个加速候选方向，逐一给出：预期提速量级（有数据标数据、无数据标"理论上/待实测"）、
   实现复杂度（改几行代码 vs 需要新起服务）、在**单机本地批量转换 3.6 万页**这个场景下的性价比排序。
4. **收敛推荐**：给出一个"本周末就能验证"的最小改动加速方案（改动量最小、预期收益最大），
   以及一个"如果本周末验证有效、值得投入一天做"的进阶方案。
